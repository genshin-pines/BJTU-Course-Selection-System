# 匿名链路迁移进度补充

更新时间：2026-06-12

## 本次完成

- 评价编辑、删除、重复评价检查已改为只认 `voter_record_id`，不再使用旧 `review.student_id` 作为权限或重复提交兜底。
- 点赞/点踩切换和投票状态查询已改为只认 `review_vote.voter_record_id` 对应的匿名记录，不再使用旧 `review_vote.student_id` 兜底。
- 举报去重和自评举报判断已改为只认 `report.reporter_record_id` 与 `review.voter_record_id`，不再使用旧 `reporter_id/student_id` 兜底。
- `voter_record.student_id` 仍保留在匿名映射表内部，用于从当前登录学生定位匿名业务身份；业务表之间的判断不直接依赖真实学生 ID。
- 已新增回归测试覆盖“旧 `student_id` 不再放行评价编辑”和“旧 `student_id` 不再触发自评举报判断”两个边界。
- 评价服务和举报服务已移除旧 `CourseMapper` 聚合刷新依赖，审核、删除、举报采纳后的统计刷新只面向 `course_instance` 与 `teacher`。
- 后台举报列表 VO 已移除真实 `reporterId` 暴露，只保留 `reporterRecordId` 和匿名展示名。
- `Review`、`ReviewVote`、`Report` Java 实体已移除旧真实身份字段入口；旧列仅保留在数据库层兼容历史数据。
- `schema.sql` 已同步当前核心模型，包含 `course_base`、`course_instance`、`course_teacher_relation`、`voter_record`，并将 `review/review_vote/report` 的匿名字段纳入新库初始化结构。

## 当前边界

- 旧表字段 `review.student_id`、`review_vote.student_id`、`report.reporter_id` 仍可作为数据库兼容列保留，但 Java 业务实体和新业务逻辑不再依赖它们。
- 历史数据必须先完成 `voter_record` 回填，否则没有匿名记录的旧评价将不能被原作者编辑或删除，这是匿名隔离收口后的预期行为。
- 后续可以在数据库迁移稳定后，再考虑将这些旧字段标记废弃、清理写入路径或从表结构中移除。
