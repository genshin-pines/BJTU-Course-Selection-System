# 编码实现报告

## 1. 项目简介

### 1.1 项目名称

北京交通大学课程评价系统（BJTU Course Review System）

### 1.2 项目背景

为帮助学生选课提供参考信息，构建一个课程评价平台。学生可以对已修课程进行匿名评价，分享学习经验、评分和考核方式；其他学生可以通过筛选和排序快速找到有价值的课程评价。管理员可以对评价内容进行审核和管理，确保平台内容质量。

### 1.3 项目目标

- 为学生提供一个匿名、真实的课程评价平台
- 通过多维度评分（综合评分、给分、授课质量、作业轻松度）帮助选课决策
- 通过标签系统实现课程快速筛选
- 通过审核机制保证评价内容质量
- 通过匿名身份映射保护学生隐私

## 2. Vibe Coding 过程

```
请帮我创建一个北京交通大学课程评价系统，技术栈要求：

  5. 举报机制
  6. 后台管理（评价审核、举报处理、课程/教师/开课实例维护、CSV导入、审计日志）

- 部署：支持 Windows 本地开发和 Ubuntu 远端部署

请先设计数据库结构，然后依次完成后端和前端开发。
```

### 2.2 开发流程

项目采用 Vibe Coding 方式，整体流程如下：

| 阶段 | 内容 | 工具/方式 |
|------|------|----------|
| 第一阶段 | 数据库设计 | AI 生成 schema.sql + 迁移脚本 |
| 第二阶段 | 后端开发 | AI 生成 Entity/Mapper/Service/Controller |
| 第三阶段 | 前端开发 | AI 生成 Vue 组件、路由、状态管理 |
| 第四阶段 | 功能完善 | 手动补充敏感词检测、匿名身份映射等 |

### 2.3 技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 匿名实现 | voter_record 表映射 | 不依赖第三方服务，数据完全可控 |
| 权限模型 | 管理员角色表 | 灵活扩展，支持未来新增角色 |
| 敏感词检测 | 后端关键字匹配 | 简单有效，无需引入外部 NLP 服务 |
| CSV 导入 | 后端逐行解析 | 不依赖重型库，适配业务逻辑 |
| 部署方式 | systemd 服务管理 | 稳定可靠，支持开机自启和自动重启 |

## 3. AI 生成后的修改

### 3.1 主要修改内容

#### 3.1.1 匿名身份映射完善

**AI 生成问题**：初始版本中匿名身份仅简单哈希学号，无法防止通过评价时间推断身份。

**修改内容**：
```
- 新增 voter_record 表，存储学号到匿名身份的映射
- AnonymityService 实现随机匿名 ID 生成
- 评价发布时自动关联 voter_record
- 投票/举报时验证匿名身份一致性
```

#### 3.1.2 敏感词检测

**AI 生成问题**：初始版本未实现内容安全审核。

**修改内容**：
```
- 在 ReviewServiceImpl 中添加敏感词检测逻辑
- 触发敏感词的评价自动设置为待审核状态
- 记录隐藏原因到审计日志
```

#### 3.1.3 课程申请功能

**AI 生成问题**：初始版本未包含学生主动申请新课程的功能。

**修改内容**：
```
- 新增 course_application 表
- CourseApplicationController 提供申请接口
- 管理员审核通过后自动创建课程/教师/开课实例
- 前端新增 CourseApplication.vue 页面
```

#### 3.1.4 评价"点没用"功能

**AI 生成问题**：初始版本仅有点赞功能，缺少负向反馈。

**修改内容**：
```
- 新增 review_vote 表支持 downvote 类型
- ReviewCard 组件新增"没用"按钮
- 后端 ReportService 扩展处理逻辑
```

#### 3.1.5 审计日志完善

**AI 生成问题**：初始版本审计日志记录不够详细。

**修改内容**：
```
- AuditLog 实体增加操作原因字段
- 所有后台操作（审核、举报处理、数据修改）均记录日志
- AdminController 增加按操作类型筛选日志
```

## 4. 存在的问题

### 4.1 已知问题

| 问题编号 | 问题描述 | 严重程度 | 状态 | 解决方案 |
|----------|----------|----------|------|----------|
| P001 | Element Plus label 弃用警告 | 低 | 待修复 | 升级 Element Plus 3 时批量修复 |
| P002 | 邮件验证码未实现频率限制 | 中 | 已知 | 后续添加 Redis 限流 |
| P003 | CSV 导入未处理大文件内存溢出 | 中 | 已知 | 增加流式读取和分批提交 |
| P004 | 敏感词库为静态列表 | 低 | 已知 | 改为可配置的词库管理 |
| P005 | 前端未实现分页懒加载 | 低 | 已知 | 后续优化长列表性能 |
| P006 | 后端未实现接口限流 | 中 | 已知 | 引入 Spring Rate Limiter |
| P007 | 单元测试覆盖率不足 | 中 | 已知 | 补充 Controller 层测试 |
| P008 | 缺少国际化支持 | 低 | 已知 | 后续添加中英文切换 |

### 4.2 技术债务

1. **敏感词管理**：当前敏感词硬编码在代码中，后续应改为可配置的管理功能。
2. **邮件限流**：验证码发送未限制频率，可能被滥用。
3. **CSV 导入优化**：大文件导入时可能 OOM，需要流式处理。
4. **前端性能**：课程列表未使用虚拟滚动，数据量大时可能卡顿。
5. **API 文档**：缺少 Swagger/OpenAPI 文档，后续可集成。
