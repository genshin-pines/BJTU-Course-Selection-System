## 1. 核心数据模型重构与匿名链路隔离

合并来源：

- 数据模型重构。
- 匿名身份物理隔离。
- 投票体系扩展中依赖匿名身份的部分。
- 设计图中剩余的 `CourseBase`、`CourseInstance`、`EvalMain`、`VoterRecord` 等核心实体。

为什么合并：

- 课程实例、评价主表、匿名记录、投票记录之间强相关。
- 如果先单独做匿名或投票，后续再拆评价主表，会产生二次迁移。
- 这部分是后续考试经验、附件、风控、排序、统计回写的共同地基。

需要完成：

- 拆分 `CourseBase` 与 `CourseInstance`。
- 建立 `CourseTeacherRelation`，支持同课异师、同师异班、不同学期课程实例。
- 将当前 `review` 逐步拆分为 `EvalMain`、`EvalTags`、`EvalExamExp`、`EvalAttachments`、`EvalVote` 等结构。
- 建立 `VoterRecord` 或等价匿名业务身份表。
- 评价和投票链路不再直接保存真实 `student_id`。
- 引入不可逆匿名标识、匿名哈希或匿名 UUID。
- 明确真实身份表和评价业务表之间的访问边界。
- 设计旧数据迁移到新模型的 SQL 或迁移程序。

预期结果：

- 系统从“简化课程评价表”升级为更接近设计类图的正式数据模型。
- 匿名评价不再只是页面展示匿名，而是链路上降低身份反推风险。
- 后续所有扩展模块都有稳定的数据基础。

## 2. 审核状态机完善与异步风控闭环

合并来源：

- 异步风控与人工复核队列。
- 设计图中匿名评价提交与异步风控流转。
- 设计图剩余特性中的 `PENDING_MANUAL`、`HIDDEN`、`ARCHIVED` 等状态。

为什么合并：

- 风控结果必须依赖评价状态机表达。
- 人工复核、违规隐藏、学期归档都属于评价生命周期管理。
- 如果只做风控检测，不做状态流转，后台无法正确承接处理结果。

需要完成：

- 完善评价状态：`PENDING_AUDIT`、`PENDING_MANUAL`、`PUBLISHED`、`HIDDEN`、`ARCHIVED`。
- 将评价提交拆分为“快速落库”和“异步检测”两个阶段。
- 实现文本违规检测、重复文本检测、恶意刷分检测、异常频率检测。
- 可疑评价自动转入 `PENDING_MANUAL`。
- 后台增加人工复核队列。
- 支持违规隐藏、恢复展示、学期归档。
- 明确只有 `PUBLISHED` 评价参与课程和教师评分统计。
- 轻量阶段可先用 Spring 异步任务，后续再替换为消息队列。

预期结果：

- 评价从提交到发布、隐藏、归档形成完整生命周期。
- 风控不再只是同步敏感词过滤，而是可流转、可复核、可追踪。
